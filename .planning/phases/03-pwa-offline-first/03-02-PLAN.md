# Plan 03-02: IndexedDB Offline Data Layer & Background Sync

**Wave:** 1 (parallel with Plan 01)
**Scope:** IndexedDB client storage, offline task CRUD, sync queue, conflict resolution

## Tasks

### T-311: IndexedDB Schema & Store
- Create `apps/web/src/domains/offline/offline-store.ts`
- Use `idb` library (lightweight promise-based IndexedDB wrapper)
- Stores: `tasks`, `framework_scores`, `pending_sync`
- Schema mirrors Drizzle task fields: id, title, description, due_date, status, updated_at, etc
- `pending_sync` store: { id, operation, table, payload, created_at, retry_count }

### T-312: Offline Task Repository
- Create `apps/web/src/domains/offline/offline-task-repo.ts`
- Implements same interface as server task repository
- CRUD operations write to IndexedDB first
- On write: also enqueue operation in `pending_sync` store
- Read path: IndexedDB data with freshness indicator (last sync timestamp)

### T-313: Background Sync Engine
- Create `apps/web/src/domains/offline/sync-engine.ts`
- Process `pending_sync` queue FIFO, one-at-a-time
- POST/PUT/DELETE to server API with retry (3 attempts, exponential backoff)
- On 409 conflict: compare `updated_at`, apply last-write-wins, log conflict event
- Emit `sync.conflict.resolved` domain event for observability
- SyncManager registration for reliable background delivery
- Fallback: 30s periodic sync when SyncManager unavailable

### T-314: Sync Queue Manager
- Queue depth warning at 100 pending operations
- Queue persistence across SW updates (IndexedDB survives)
- Stale detection: operations older than 24h flagged for review
- Deduplication: collapse multiple updates to same task into latest

### T-315: Tests
- IndexedDB store CRUD (using fake-indexeddb in tests)
- Sync engine processes queue in order
- Conflict resolution applies last-write-wins correctly
- Queue deduplication collapses updates
- Full round-trip: offline write → queue → sync → server

**Total:** 5 tasks
